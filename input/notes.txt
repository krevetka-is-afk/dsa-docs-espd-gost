Рабочие заметки для генератора term-paper.

Источник правды для документации: актуальное ТЗ из `docs/technical-specification`,
код, README, docs/api_scheme.md, docs/architecture/contracts/*.json и tests.
Разделы ПЗ, ПМИ и РО должны использовать ТЗ как источник формальных требований,
а код и контракты -- как источник проверяемых технических деталей.

Реализованный scope:
- Django + DRF backend в src/web.
- HTML frontend на Django templates: auth, verify email, catalog, project
  detail, project form, moderation queue, applications, profile.
- Роли: student, customer, cpprp/staff.
- Локальная регистрация/логин по email/password с подтверждением email
  шестизначным кодом. SMTP/console backend настраивается через env.
- API token endpoint блокирует выдачу токена до подтверждения email.
- Каталог проектов, модель ЭПП/Project, модерация, заявки, избранное,
  интересы, рекомендации.
- Импорт ЭПП из XLSX через API и команду CLI.
- Role-based account endpoints для student/customer/cpprp кабинетов.
- django-unfold admin для улучшенного администрирования.
- Faculty domain: синхронизация преподавателей, публикаций, курсов,
  matching научных руководителей проектов с FacultyPerson.
- Outbox API доставки событий для внешних потребителей ML, graph и faculty.
- ML reference service на FastAPI: search, recommendations, reindex,
  outbox-driven index, режимы semantic и stub-heuristic.
- Graph service на FastAPI + Neo4j: projector по outbox events.
- Production/staging Docker Compose, Nginx, backup/restore, env sync scripts.

Ограничения и уточнения, которые нужно честно отражать в ПЗ/ПМИ/РО:
- README, базовые docs и ТЗ актуализированы под согласованный состав проекта: текущий
  основной режим аутентификации - локальный login/register с email
  verification; header SSO adapter есть как integration layer, но production
  SSO edu.hse.ru не является приемочным требованием без отдельного
  подтверждения.
- Текущий integration surface - REST/JSON + outbox API, а не обязательная
  Kafka/SSE реализация.
- ML service является reference implementation; при отсутствии semantic
  ответа веб-приложение переходит к резервному ранжированию по ключевым словам.
- Полный release gate требует доступный Neo4j. Локальный ALLOW_LOCAL_GRAPH_SKIP
  не должен считаться финальным приемочным прогоном.
- Длина комментариев модерации должна соответствовать backend: 100 символов
  для отклонения проекта, 50 символов для отклонения заявки и инициативной
  темы.
- Для финальных документов нужны скриншоты: auth, verify email, catalog,
  project detail, project form, moderation queue, applications, profile,
  admin/unfold, API docs.

Рекомендуемая формулировка архитектуры:
Модульный монолит Django + DRF + Django templates как основной источник данных,
отдельные сервисы для ML и graph, зеркало и сопоставление faculty в web,
синхронизация внешних потребителей через outbox API доставки событий. Целевое
хранилище - PostgreSQL, локальный release gate - SQLite, графовая проекция - Neo4j.
